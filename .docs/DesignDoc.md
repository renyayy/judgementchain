# 概要 (Abstract/Summary)
プロジェクト全体の要約です。レビュー者はここを読むだけで、何を作ろうとしているのか、なぜそれが必要なのかを数文で素早く理解できます。
# 背景と動機 (Background/Motivation)
「なぜ今、このプロジェクトが必要なのか？」「現状のシステムにはどんな課題があるのか？」を説明します。レビュー者がプロジェクトの文脈を理解し、適切なフィードバックをするために不可欠なセクションです。
# 目標 (Goals)
このプロジェクトで「達成したいこと」を具体的かつ明確にリストアップします。成功・失敗が判定可能な目標を設定できると、より効果的です。
# 非目標 (Non-Goals)
このプロジェクトで「やらないこと」を意図的に定義します。これにより、プロジェクトの範囲が不必要に拡大すること（スコープクリープ）を防ぎ、議論の焦点を絞ることができます。
# 提案する設計 (Proposed Design)
ここでは、ドキュメントの核となる技術的な側面を詳述します。目標達成のための具体的な方法を、アーキテクチャ図やシーケンス図などを用いて視覚的に説明し、システムの全体像を読者が容易に把握できるようにします。レビューを円滑に進めるため、必要に応じてサブセクションを設けて内容を整理します。

アーキテクチャ: システム全体の構造、主要なコンポーネント、およびそれらの相互関係について。
データモデル: 扱うデータの具体的な構造や内容について。
APIデザイン: 各コンポーネントの連携動作に関する現時点での設計について。
ユーザーインターフェース (UI) の考慮事項: ユーザーに見せる部分について
セキュリティとプライバシー: 考えられるリスクと、それらに対する現時点での対策について
パフォーマンスとスケーラビリティ: 想定されるアクセス数やデータ量と、それらに対する現時点での対策について
テスト戦略: 品質保証をどのように行うかの現時点での戦略について
# 代替案の検討 (Alternatives Considered)
提案した設計以外に検討した他のアプローチと、「なぜそれらを選ばなかったのか」をトレードオフと共に説明します。これにより、提案する設計が客観的に見て最適であることの説得力が増します。
# 未解決の課題 (Open Questions)
設計段階でまだ答えが出ていない問題や、さらなる議論が必要な点をリストアップします。これを明確にすることで、見落としを防ぎ、チーム全体で解決策を考えるきっかけになります。
# 今後の作業 (Future Work)
このプロジェクトの次のステップや、将来的に追加したい機能などを記述します。
